# Tarot LangGraph Reading Workflow Design

**Date:** 2026-07-16
**Status:** Approved
**Primary rollout:** Question-based one-card tarot
**Future reuse:** Daily one-card and three-card tarot

## Goal

Replace the current monolithic tarot prompt path with a thin, deterministic LangGraph workflow that separates question understanding, tarot grounding, reading generation, validation, retry policy, and persistence.

The workflow must improve Korean reading quality without turning the product into a chat agent. It must support both preset questions and one-shot free-form questions, remain grounded in the selected card and orientation, and never return a low-quality local fallback as if it were a completed reading.

## Product Principles

- A free reading must be good enough to demonstrate the value of paid tarot content.
- The reading answers the user's actual question before explaining the card.
- Tarot symbolism remains visible, but the prose should read as natural Korean rather than translated keyword copy.
- The system may interpret a card, but it must not invent specific events, amounts, dates, hidden thoughts, or guaranteed outcomes.
- A failed generation does not consume a free use, rewarded-ad unlock, or paid entitlement.
- The selected card, orientation, and question remain stable across repair, regeneration, and user retry.
- The workflow is request-scoped. It has no conversation memory and asks no follow-up questions.

## Why LangGraph

LangGraph is used as a workflow coordinator, not as the tarot interpretation engine. The product needs deterministic stages and explicit failure branches, but does not need an autonomous agent.

Use:

- typed request-scoped state
- deterministic nodes
- conditional edges
- bounded repair or regeneration
- observable node timing and failure categories

Do not use:

- checkpointers or `thread_id`
- conversation persistence
- interrupts or human-in-the-loop execution
- autonomous tools
- multi-agent collaboration
- unbounded evaluator loops
- RAG for the fixed 78-card corpus

## Architecture

```text
START
  -> validate_request
  -> resolve_question
  -> apply_scope_policy
  -> build_card_grounding
  -> generate_reading
  -> validate_reading
       |-- pass ------------------------------> finalize
       |-- style_error -> repair_reading ----> validate_once
       |-- grounding_error -> regenerate ----> validate_once
       `-- provider_error --------------------> fail
  -> END
```

Repair and regeneration share one retry budget. A request receives at most one automatic post-generation LLM retry. A second invalid result ends in a retryable product failure state.

After the graph returns a validated final reading, the API route applies persistence and usage policy. Persistence is deliberately outside the graph so transport and entitlement side effects remain in the existing route boundary.

Preset question resolution does not call an LLM. Free-form question resolution uses one small structured-output call. The main reading is generated by the configured high-quality tarot model.

## Responsibility Boundaries

### API Route

The API route owns transport and product policy, not tarot interpretation.

- authenticate the user or resolve the guest session
- validate spread, date, card selection, orientation, and unlock method
- resolve preset keys against server-owned canonical content
- enforce free-question length and supported category keys
- check usage before generation
- invoke the workflow
- persist only a successful final reading
- increment usage only after successful generation and persistence policy checks
- return a retryable product error without technical provider details

Client-provided labels, preset question text, card meanings, and interpretation metadata are not authoritative.

### LangGraph Workflow

The graph owns stage ordering, state transitions, retry budget, and failure classification. It does not contain card meanings or user-facing prose templates.

### Question Resolver

The resolver determines what the user is asking. It does not write the tarot reading.

- Preset question: load the canonical question contract by server-resolved state and question keys.
- Free-form question: classify the input into a bounded structured `QuestionBrief`.
- Classification failure: use the selected category's default contract while preserving the original question as context.
- Mixed question: keep one primary domain and zero or more secondary domains.
- Vague statement: use a reflective answer mode from the selected category instead of asking a follow-up.

The free-form input is untrusted data. It is length-limited, quoted as user content, and cannot provide system instructions or override policy.

### Scope Policy

The scope policy adjusts what a reading is allowed to claim without changing the user's topic.

- third-party feelings: describe observable relationship dynamics and the user's perspective; do not assert hidden thoughts as fact
- future questions: describe pressures, likely direction, and decision criteria; do not guarantee outcomes or dates
- medical, legal, and financial questions: keep the response reflective and non-authoritative
- supernatural communication: frame the result as personal meaning or memory, not verified contact
- crisis content: do not continue with ordinary divination; return the dedicated safety response path

### Card Knowledge Source

Canonical card content is the factual source for generation. The server resolves cards by stable card key and never accepts card meanings from the client.

The target shared content boundary is `@manyang/content`. Reading-specific profiles should be a separate export so UI code can use normal card content without importing prompt orchestration. Until the shared content extraction is complete, the workflow must still consume one canonical server-resolved source rather than duplicating meanings in prompt code.

### Grounding Builder

The grounding builder is deterministic. It selects only the evidence relevant to the chosen orientation and question domain.

It sends:

- selected card and orientation
- selected orientation meaning only
- core card dynamic
- human behavior or emotional pattern
- relevant domain application
- two or three actual visual symbols from this deck
- healthy and distorted expressions
- central tension
- prohibited unsupported claims

It does not send:

- the opposite orientation meaning
- a separate fixed `cardMessage` that duplicates the conclusion
- fixed phrase palettes
- generic suit and rank copy when card-specific evidence exists
- unrelated domain meanings

### Reading Generator

The generator connects the question brief and card grounding. A single high-quality model call produces both a hidden interpretation plan and user-facing prose. This avoids a separate planning call while preserving an inspectable semantic structure.

The generator does not classify the question, choose a different card, or decide usage policy.

### Validator

The validator decides whether a draft is acceptable. It does not silently rewrite a failed reading.

Deterministic checks run first. A short semantic evaluator may run only when a meaning-level decision cannot be made reliably from structured fields and local rules. Genericness and broad style calibration primarily belong in offline evaluation rather than an unconditional production judge call.

### Persistence

Persistence receives only a validated final reading. Failed drafts and internal interpretation plans are not stored as user readings.

## Workflow State

```ts
type TarotReadingWorkflowState = {
  request: ValidatedTarotRequest;
  questionBrief?: QuestionBrief;
  scopeDecision?: TarotScopeDecision;
  grounding?: TarotCardGrounding;
  draft?: TarotReadingDraft;
  validation?: TarotReadingValidation;
  retryMode?: "repair" | "regenerate";
  attemptCount: number;
  finalReading?: TarotGeneratedReading;
  failure?: TarotReadingFailure;
};
```

The state exists only for one request. There is no LangGraph checkpointer, conversation transcript, or cross-request memory.

## Question Model

### Preset Question Contract

The visible catalog keeps state and question keys. Each question maps to a stable intent key rather than embedding a custom prompt.

```ts
type TarotQuestionContract = {
  intentKey: string;
  primaryDomain: TarotQuestionDomain;
  askType: TarotQuestionAskType;
  directAnswerTarget: string;
  requiredElements: readonly TarotAnswerElement[];
  allowedModes: readonly TarotInterpretationMode[];
  preferredClosing: TarotClosingMode;
  forbiddenClaims: readonly TarotForbiddenClaim[];
};
```

The 30 preset questions should map to roughly 10 to 15 reusable intent contracts. They must not become 30 independent prompt blocks.

Representative contracts:

- hidden emotion: name the feeling, explain why it is avoided, and show how it appears in ordinary behavior
- relationship emotional current: name the relationship atmosphere, explain what creates it, and describe how it is expressed without claiming private thoughts
- blocked work: identify the bottleneck, its maintaining behavior, and the first area to reconsider
- practical resources: identify a realistic area to inspect without inventing a transaction, amount, document, or date
- choice criterion: identify the trade-off, decision criterion, and responsibility the user would accept
- daily focus: identify one priority and explain why this card points to it
- supportive energy: identify a usable attitude or capacity and where it would help

### Free-Form Question Brief

Free-form questions are not forced into the nearest preset. The resolver combines a reusable base contract with extracted, question-specific context.

```ts
type QuestionBrief = {
  source: "preset" | "freeform" | "category_default";
  originalQuestion: string;
  primaryDomain: TarotQuestionDomain;
  secondaryDomains: TarotQuestionDomain[];
  askType: TarotQuestionAskType;
  subject: string;
  directAnswerTarget: string;
  emotionalContext?: string;
  timeHorizon: "now" | "near" | "general";
  answerMode: TarotAnswerMode;
  sensitivity: "normal" | "bounded" | "crisis";
  needsReframing: boolean;
  requiredElements: TarotAnswerElement[];
  forbiddenClaims: TarotForbiddenClaim[];
};
```

Structured classification fields use enums or short bounded strings. The resolver must preserve the user's subject and concern instead of replacing them with generic category copy.

## Card Interpretation Profile

Do not build a 78-card by 30-question matrix. Build one card profile layer, one question contract layer, and compose them through an interpretation mode.

```ts
type TarotCardInterpretationProfile = {
  cardKey: string;
  coreDynamic: string;
  humanPattern: string;
  emotionalTone: readonly string[];
  sceneEvidence: readonly TarotSceneEvidence[];
  uprightExpression: TarotOrientationExpression;
  reversedExpression: TarotOrientationExpression;
  healthyExpression: string;
  distortedExpression: string;
  tensions: readonly TarotCardTension[];
  domainApplications: Readonly<Record<TarotQuestionDomain, string>>;
  contentVersion: string;
};

type TarotSceneEvidence = {
  id: string;
  visual: string;
  interpretiveUse: string;
};

type TarotOrientationExpression = {
  movement: string;
  behavior: string;
  caution: string;
};
```

Reversed meanings are not automatic opposites. They may represent blocked, excessive, distorted, delayed, hidden, or internalized forms of the card's core dynamic.

All visual evidence must match the actual deck image. Generic Rider-Waite-Smith symbolism may be used only when it is visibly present in this deck.

## Interpretation Modes

The interpretation mode describes how the card answers the question. It is not selected randomly for stylistic variety.

```ts
type TarotInterpretationMode =
  | "direct_answer"
  | "hidden_cause"
  | "inner_conflict"
  | "question_reversal"
  | "developing_process"
  | "boundary";
```

- `direct_answer`: the card directly names the condition, feeling, or priority
- `hidden_cause`: the card explains what maintains the visible problem
- `inner_conflict`: the card shows two competing needs or expressions
- `question_reversal`: the card corrects a weak assumption in the question
- `developing_process`: the card shows a condition still changing rather than a settled answer
- `boundary`: the card clarifies what can and cannot responsibly be inferred

Each question contract limits the modes that make sense for that question. The generator selects one mode and states one clear thesis before writing the final prose.

## Generation Contract

The structured output keeps internal evidence separate from display copy.

```ts
type TarotReadingDraft = {
  interpretation: {
    mode: TarotInterpretationMode;
    thesis: string;
    usedSymbolIds: string[];
    answeredTarget: string;
  };
  display: {
    title: string;
    keywords: string[];
    overview: string;
    cardReadings: TarotGeneratedCardReading[];
  };
};
```

For question one-card readings:

- the first sentence directly answers `directAnswerTarget`
- the explanation connects the answer to the card's dynamic and selected orientation
- one or two actual symbols support the interpretation instead of decorating it
- the final sentence serves the question contract; it does not always become a criterion, action, or abstract realization
- sentence count and rhetorical order may vary
- `cardReadings` remains empty for transport compatibility unless a later API version changes it

The model must not receive a fixed four-sentence template. Semantic requirements are fixed; sentence shapes are not.

## Validation

### Hard Validation

- structured output matches the schema
- spread-specific fields and card positions are exact
- selected card and orientation are unchanged
- every `usedSymbolId` exists in the supplied grounding
- required question elements are represented
- internal field names are absent from display copy
- unsupported dates, amounts, transactions, diagnoses, legal conclusions, or private-thought certainty are rejected
- title, keywords, overview, and card readings satisfy length and count bounds
- repair and regeneration cannot exceed the shared retry budget

### Quality Validation

Evaluation should score:

- directness: does the reading answer the actual question early?
- card grounding: could the interpretation be traced to this card and orientation?
- question specificity: would the text stop making sense if the question changed?
- card specificity: would the text stop making sense if the card changed?
- natural Korean: is the prose idiomatic and understandable aloud?
- symbolic value: does tarot imagery contribute meaning rather than keyword decoration?
- boundedness: are uncertainty and unsupported specifics handled correctly?
- non-repetition: does the result avoid recurring formulaic endings and phrase palettes?

Card-swap and question-swap checks belong in the evaluation suite. A reading that still works after swapping either input is too generic.

## Retry and Failure Policy

The validation result determines one bounded retry path:

- `style_error`: repair the same thesis and evidence; do not reinterpret the card
- `grounding_error`: discard the draft and regenerate from the original question brief and grounding
- `safety_error`: discard the draft and regenerate with the tightened scope decision when safe to do so
- `provider_error`: return a retryable failure without generating substitute copy

After one repair or regeneration, validate once more. If it still fails:

- do not return a local dictionary or template fallback
- keep the selected card, orientation, and question
- do not persist the failed draft
- do not consume usage or unlock entitlement
- offer `Retry reading` with the same card and question
- avoid exposing model, timeout, schema, or provider details in the UI

## Model Call Budget

Normal preset path:

```text
main reading generation
```

Normal free-form path:

```text
small structured question resolution
-> main reading generation
```

Exceptional path:

```text
normal path
-> at most one repair or regeneration call
```

A semantic judge is conditional, not a required call on every production request. Repeated quality comparison should run offline and against sampled, privacy-safe production telemetry.

## Privacy and Provider Settings

- Send only the information needed for the current one-shot reading.
- Do not send previous readings or conversation history.
- Treat free-form text as user data, never as instructions.
- Use `store: false` for OpenAI Responses API calls.
- Do not log raw free-form questions in operational logs.
- Persist user-visible questions only through the existing product reading record policy.
- Keep evaluator datasets synthetic or explicitly curated unless user data has a separate consent policy.

## Observability and Versioning

Every workflow result should expose internal operational metadata without leaking it into display copy:

```ts
type TarotWorkflowTelemetry = {
  workflowVersion: string;
  promptVersion: string;
  cardProfileVersion: string;
  questionContractVersion: string;
  model: string;
  questionSource: "preset" | "freeform" | "category_default";
  attemptCount: number;
  retryMode?: "repair" | "regenerate";
  nodeDurationsMs: Record<string, number>;
  providerCallCount: number;
  outcome: "success" | "failure";
  failureReason?: string;
};
```

Track latency, timeout, invalid response, validation reason, retry success, and model usage. Do not include raw private question text in logs.

## Code Boundaries

Target backend modules:

- `backend/src/services/tarot-question-resolver.ts`: preset and free-form question resolution
- `backend/src/services/tarot-scope-policy.ts`: bounded and crisis scope decisions
- `backend/src/services/tarot-grounding-builder.ts`: orientation- and domain-specific evidence selection
- `backend/src/services/tarot-reading-prompt.ts`: generation and retry prompt contracts only
- `backend/src/services/tarot-reading-validator.ts`: deterministic and conditional semantic validation
- `backend/src/services/tarot-reading-workflow.ts`: LangGraph state and conditional edges
- `backend/src/services/llm-tarot-reading.ts`: stable public service facade and provider error mapping

Target content and contract boundaries:

- public state, question, card, and transport keys in shared contracts/content
- server-owned intent contracts and safety policy in backend services
- canonical 78-card interpretation profiles in the shared tarot content source, exported separately from UI-oriented card data

The existing frontend API route remains responsible for authentication, usage, canonical request resolution, persistence, and HTTP responses. It should call the stable `generateTarotReadingForUser` facade rather than importing LangGraph nodes directly.

## Testing Strategy

### Unit Tests

- all preset keys resolve to canonical intent contracts without an LLM call
- free-form structured results reject unknown enums and oversized extracted fields
- classifier failure uses the selected category default while preserving original input
- mixed-domain questions retain one primary and supported secondary domains
- scope policy constrains third-party certainty, future certainty, and professional advice
- all 78 cards have complete, card-specific profile data before production rollout
- reversed profiles describe a valid transformation rather than a mechanical opposite
- every visual symbol ID is unique within its card and matches available grounding
- grounding contains only the selected orientation and relevant domain data
- output parsing rejects unknown symbol IDs and internal field leaks
- style, grounding, safety, and provider failures route to the correct edge
- repair and regeneration share a maximum retry count of one
- failed workflows do not persist or consume usage

### Integration Tests

- preset question request follows the no-classifier path
- free-form request follows classifier then generator
- classifier timeout follows the category-default path
- generation timeout returns the retryable failure contract
- repair preserves card, orientation, question brief, thesis, and evidence
- regeneration preserves card, orientation, and question but may replace the thesis
- successful retries persist exactly once
- OpenAI requests disable provider-side storage

### Evaluation Suite

Build a curated matrix containing:

- all preset intent contracts
- clear, vague, mixed-domain, colloquial, and adversarial free-form inputs
- major, minor, court, ace, and numbered cards
- upright and reversed orientations
- relationship, work, money, self, choice, and daily domains
- high-risk certainty prompts

Run repeated generations per case and compare models using a human-calibrated rubric. Reports must include scores, latency, failure rate, and raw side-by-side outputs. Model choice should not be based on a single example.

## Rollout

### Phase 1: Interpretation Pilot

- author 12 deliberately varied card profiles
- define all reusable question contracts
- implement the graph behind a feature flag for question one-card readings
- run structured and human A/B evaluation against the current path
- revise the profile schema and rubric before multiplying content

The pilot set should include major and minor cards, a court card, an ace, several numbered cards, and both straightforward and ambiguous reversals.

### Phase 2: Full Card Grounding

- complete and review all 78 card profiles
- add exact-deck visual evidence checks
- pass completeness and card-swap evaluation thresholds
- enable the workflow for internal or sampled production traffic

No production-wide switch occurs with partial generic minor-arcana profiles.

### Phase 3: Question Tarot Rollout

- enable preset question traffic
- enable free-form question traffic after classifier and category-default tests pass
- monitor latency, validation failures, retries, and reading-quality samples
- compare conversion and repeat-use behavior without lowering the free reading model quality

### Phase 4: Reuse

- migrate daily one-card through an appropriate daily contract
- add three-card synthesis contracts and card-to-card relationship grounding
- keep the same workflow facade and failure policy while using spread-specific generation contracts

## Alternatives Rejected

### Keep expanding one prompt

Adding more prohibited phrases and question-specific prompt blocks has already increased repetition and made behavior harder to reason about. It does not solve weak card data or semantic validation.

### Write every card-question combination

A 78 by 30 matrix is expensive to maintain, prevents free-form expansion, and creates inconsistent revisions. Composing card profiles, question contracts, and interpretation modes scales better.

### Full conversational agent

The product is not a chat flow. Memory, follow-up tools, and autonomous decisions add complexity without improving a one-shot reading.

### RAG over tarot documents

The 78-card corpus is fixed and small enough for curated structured data. RAG would add retrieval uncertainty without addressing prose quality.

### Local fallback reading

A dictionary or template fallback would present a visibly lower-quality result after the user has invested attention or watched an ad. Failure with a preserved retry is the approved product behavior.

## Completion Criteria

- preset and free-form questions converge on the same typed `QuestionBrief` contract
- every production card has reviewed, exact-deck grounding for both orientations
- the generator receives only relevant orientation and domain evidence
- the display reading directly answers the question and cites real card imagery naturally
- hard validation, bounded retry, and failure-without-consumption behavior are covered by tests
- no local fallback can be returned as a successful reading
- no LangGraph persistence or conversation state is enabled
- provider calls disable response storage where supported
- model and prompt changes can be compared through repeatable, scored evaluation
- question one-card rollout meets agreed quality, latency, and failure thresholds before broader tarot migration
