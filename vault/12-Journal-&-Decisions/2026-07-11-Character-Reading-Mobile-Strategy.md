# 2026-07-11 Character Reading Mobile Strategy Decision

> 꿈 중심 웹 서비스에서 캐릭터·타로 중심 감성 리딩 서비스와 별도 React Native 앱 전략으로 확장한다.

---

## 배경

기존 마냥은 꿈 기록과 꿈 해몽을 중심으로 아침·밤 기록, 발자국, 아카이브를 리텐션 장치로 사용했다. 콘텐츠 확장 논의에서 사용자가 꿈 해몽보다 타로를 더 가볍고 자주 소비하며, 마냥의 고양이 캐릭터 자체도 방문 동기가 될 가능성이 크다고 판단했다.

기존 문서는 고양이를 시각 테마로만 제한하고 모바일 앱을 먼 단계로 두었다. 이 결정은 해당 장기 방향을 수정한다.

## 결정

1. 마냥을 **고양이가 읽어주는 감성 운세·리딩 서비스**로 확장한다.
2. 타로를 주 소비 콘텐츠로, 꿈·감정 기록을 개인화 자산으로 사용한다.
3. 한 마리의 주 동반자와 깊은 관계를 쌓되 다른 고양이의 에피소드와 수집도 제공한다.
4. 관계는 무료 활동으로 성장하고 결제로 구매할 수 없다.
5. 달조각은 앱 전용 구매·소비 재화로 사용한다.
6. 웹은 무료 체험·SEO·앱 설치 전환을, 앱은 관계·수집·결제를 담당한다.
7. 앱은 Capacitor 래퍼가 아니라 Expo·React Native 별도 클라이언트로 제작한다.
8. 웹과 앱 UI는 분리하고 제품 규칙·콘텐츠·API 계약·서버를 공유한다.

## 대체되는 이전 방향

- 고양이를 시각 테마와 스킨으로만 제한하던 장기 방향
- 꿈 기록 누적만을 주 리텐션 근거로 두던 방향
- 웹과 모바일의 UI 코드를 하나로 유지하려는 방향
- 웹 결제를 먼저 구현한 뒤 앱으로 이전하는 방향

다만 다음 기존 원칙은 유지한다.

- 고양이별로 리딩 정확도와 안전 기준을 다르게 만들지 않는다.
- 기본 꿈 기록과 기본 꿈 해몽은 무료다.
- 불안 자극형 과금과 확률형 뽑기를 사용하지 않는다.
- AI 호출, 결제, 달조각 장부는 서버에서 처리한다.

## 결과

- 캐릭터 콘텐츠와 앱 전용 경험 제작량이 증가한다.
- Next.js 웹과 React Native 앱의 UI를 별도로 관리해야 한다.
- 공통 패키지와 API 인증 경계를 먼저 정리해야 한다.
- 실제 결제 검증은 앱 출시 이후 가능하며 웹에서는 앱 전환 의향을 먼저 측정한다.
- 기존 `Cat-Reader-Access-Strategy`와 `System-Architecture`는 역사적 MVP 기준으로 남기고 새 문서를 우선한다.

## 열린 결정

- 달조각 팩과 Moon Pass 가격
- 개인화 데이터 사용 범위
- 캐릭터 에피소드 운영 주기
- 앱 출시 일정과 최소 기능
- 세부 측정 지표와 성공 기준
